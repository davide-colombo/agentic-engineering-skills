# Manual Curation Artifact Packaging

## Purpose

Determine whether a curation or manual-review deliverable is assembled safely from internal pipeline outputs without overwriting the source artifacts, includes the intended inventory, separates review copy from source, names files reproducibly, carries an audit-friendly summary, preserves provenance, omits temporary or hidden system files, hides no private machine paths, and can be regenerated by an inspectable packaging command.

## When to use

Use when assembling a deliverable for a domain reviewer, curator, collaborator, or stakeholder, when packaging an intermediate review artifact for downstream curation, or when auditing a previously assembled deliverable before it is shared. Use alongside [`manifest-checksum-and-provenance`](../manifest-checksum-and-provenance/SKILL.md) when the deliverable includes a manifest, and alongside [`excel-workbook-integrity`](../excel-workbook-integrity/SKILL.md) when the deliverable includes Excel workbooks.

## When not to use

Do not use as authorization to share, publish, or transmit the deliverable. Do not substitute it for repository-state, code-review, configuration, or domain validation. Do not use it to evaluate the scientific correctness of contents; defer to [`scientific-data-integrity-audit`](../scientific-data-integrity-audit/SKILL.md). Do not use it to evaluate transfer integrity; defer to [`data-transfer-and-integrity`](../data-transfer-and-integrity/SKILL.md).

## Required inputs

Obtain from the current prompt and authoritative project material:

- Task objective, selected task mode, and allowed scope.
- The intended audience and review purpose.
- The source artifact set and the boundary between source and review copy.
- The packaging output location and naming policy.
- Permission boundaries: whether new files may be created and whether source artifacts may be modified.

If the source set, audience, review purpose, or packaging policy cannot be established, stop and report.

## Optional inputs

Use when supplied or discoverable within scope:

- Prior deliverables for the same audience.
- Project-wide deliverable naming convention.
- Reviewer-side tooling that imposes file-format expectations.
- Existing summary tables, audit logs, or operator notes.

Treat absent evidence as uncertainty, not as proof of packaging correctness.

## Task modes

Choose exactly one mode and state it in the report:

- `DELIVERABLE_DESIGN_REVIEW`: Evaluate a proposed deliverable layout before assembly.
- `DELIVERABLE_ASSEMBLY_PLAN`: Produce an inspectable plan for assembling the deliverable without performing the copy.
- `DELIVERABLE_AUDIT`: Inspect an assembled deliverable against the intended inventory and packaging contract.
- `PACKAGING_COMMAND_AUDIT`: Inspect the script or command used to assemble the deliverable for reproducibility.

A mode changes audit emphasis; it does not authorize copying, deletion, modification of source artifacts, or transmission of the deliverable.

## Deliverable inventory

For the deliverable's content set, document:

- Each required file or directory and its declared role.
- Each optional file or directory and the condition under which it is included.
- File counts and (where contractual) byte sizes or checksums.
- Sheet-level or section-level expectations when applicable.
- Tables, sequence files, plots, READMEs, and accompanying summary artifacts.

A deliverable without an explicit inventory is unsafe to assemble or audit. Record it.

## Intended audience and review purpose

State explicitly:

- The receiver category (curator, collaborator, stakeholder) without naming individuals unless required by policy.
- The review purpose (classification review, manual curation, sanity check, sign-off).
- Reviewer-side constraints (no internal paths, no proprietary identifiers, no machine-specific naming).
- Whether the deliverable is intended as a one-time review or a recurring delivery.

The audience and purpose constrain naming, contents, and what redaction is required.

## Source artifact versus review-copy separation

Maintain a strict separation between source artifacts and review copy:

- The review copy lives at a distinct path and never overlaps the source root unless explicitly authorized.
- The review copy is constructed from copies, not by renaming or moving source artifacts.
- The source artifacts are not modified during assembly.
- The review copy is regenerable from source; the source is not derivable from the review copy.

A packaging operation that mutates source is not packaging; it is overwriting. Stop and report.

## No overwrite of original source artifacts unless explicitly authorized

Confirm before any non-read action:

- The current prompt explicitly authorizes any modification of source artifacts.
- When no authorization is present, the operation is constrained to read source and write review copy.
- Atomic-copy semantics apply: the review copy becomes visible only when complete.
- A failed assembly leaves the review-copy area in a state that the next run can clean up reproducibly.

## Archive contents

When the deliverable is packaged as an archive:

- Define archive format (zip, tar, tar.gz, or other), compression behavior, and root layout.
- Confirm archive entries match the intended inventory.
- Confirm path safety: no absolute paths, no parent-directory traversal, no unsafe symlinks.
- Confirm permissions, executability, and metadata follow the project's policy.

Cross-reference [`data-transfer-and-integrity`](../data-transfer-and-integrity/SKILL.md) for in-depth archive and packaging audit when the archive is transferred.

## File naming

For each file in the deliverable:

- Apply the deliverable naming convention (date format, identifier prefix, version, audience-appropriate strings).
- Reject embedded private machine paths, host names, user names, or environment-specific strings.
- Confirm names are stable under reassembly with identical inputs.
- Confirm names sort in a way that a reviewer can navigate.

## Counts

State and verify counts:

- File counts per top-level category.
- Sheet counts when workbooks are present.
- Sequence counts when sequence artifacts are present.
- Row counts when tables are present.

Counts must be reproduced by an inspectable command, not by reviewer memory.

## Summary and audit tables

When the deliverable includes a summary or audit table:

- Document the columns, their role, and the source they were computed from.
- Confirm the table reflects the actual inventory and not a stale prior assembly.
- Cross-reference [`output-contract-and-table-schema-audit`](../output-contract-and-table-schema-audit/SKILL.md) for schema integrity.

## Provenance

Record provenance for the deliverable itself:

- The packaging command or script and its version.
- The source artifact set identity (paths, manifest, or hash).
- The packaging operator, time, and tool environment.
- Cross-reference [`manifest-checksum-and-provenance`](../manifest-checksum-and-provenance/SKILL.md) when a manifest accompanies the deliverable.

A deliverable without provenance cannot be reproduced or audited later.

## No temporary files

For every file in the deliverable, confirm:

- No `.tmp`, `.part`, `.swp`, `.bak`, `.lock`, or editor temporary files are included.
- No partially written files from an interrupted assembly are included.
- No build-system artifacts unrelated to the review purpose are included.

## No hidden system files where inappropriate

Inspect for system-generated hidden files:

- `.DS_Store`, `Thumbs.db`, `__MACOSX/`, `.Spotlight-V100/`, `desktop.ini`, and similar.
- `.git`, `.svn`, `.hg`, `.idea`, `.vscode`, or other tool directories unless explicitly required.
- `.env`, `.npmrc`, `.netrc`, or credential-bearing dotfiles.

Hidden files leak information about the producing environment. Recommend exclusion unless project policy explicitly includes them.

## No private machine paths in deliverables unless intentionally included

For text content within the deliverable:

- Inspect summary tables, READMEs, audit logs, and embedded metadata for absolute paths, machine names, user names, or environment-specific strings.
- Recommend redaction or restructuring when present.
- Document explicit exceptions where audit material legitimately requires private path retention.

## README and summary clarity

When the deliverable includes a README or summary document:

- State the deliverable purpose, audience, version, contents, and packaging command.
- State the meaning of each top-level file or directory.
- State what is in scope and out of scope.
- State known limitations and unresolved items.
- Cross-reference [`evidence-citation-discipline`](../evidence-citation-discipline/SKILL.md) when claim-level wording matters.

A README that is missing or generic forces reviewers to reconstruct context.

## Reproducibility of packaging command or script

Confirm that the deliverable is regeneratable:

- The packaging command or script is preserved and version-controlled.
- The command's inputs (source paths, parameters, exclusion rules) are stated.
- A second invocation under the same inputs produces an identical deliverable, or the differences are documented and explained.
- Manual steps (if any) are listed explicitly with their dependencies.

A deliverable that cannot be regenerated is a one-off. Record this and report it.

## Audit procedure

1. Restate the objective, selected task mode, scope, audience, review purpose, and source-versus-review-copy boundary.
2. Inspect the deliverable inventory against the intended inventory.
3. Inspect source-versus-review-copy separation and confirm no source modification has occurred.
4. Inspect the archive layout when an archive is part of the deliverable.
5. Inspect file naming, counts, summary tables, and provenance.
6. Inspect for temporary files, hidden system files, and embedded private paths.
7. Inspect the README or summary document for clarity and accuracy.
8. Inspect the packaging command or script for reproducibility.
9. Classify findings, record commands actually run and evidence unavailable, and select exactly one verdict.

Do not claim a deliverable is ready unless each inventory element and packaging requirement has been inspected.

## Stop-and-report triggers

Stop before sharing, transmitting, or accepting a deliverable when:

- Audience, review purpose, or source-versus-review-copy boundary is ambiguous.
- A source artifact has been modified during assembly.
- The archive layout includes unsafe paths, hidden system files, temporary files, or unintended content.
- File names embed private machine paths or environment-specific strings.
- Counts, summary tables, or manifests disagree with actual inventory.
- The README is missing or does not state purpose, audience, contents, and packaging command.
- The packaging command cannot regenerate the deliverable.
- Required evidence cannot be inspected and the gap prevents a reliable verdict.

Report the trigger, affected element, evidence, and minimum decision or authorization required.

## Output contract

Produce these sections:

1. `VERDICT`: exactly one verdict from the vocabulary below and a one-sentence rationale.
2. `MODE AND SCOPE`: selected task mode, audience, review purpose, source-versus-review-copy boundary, allowed scope, and exclusions.
3. `INVENTORY`: required files, optional files, counts, and roles.
4. `SOURCE SEPARATION`: evidence that source artifacts were not modified and that review copy is at a distinct path.
5. `ARCHIVE LAYOUT`: format, root layout, path safety, and entry list; write `none` when no archive is part of the deliverable.
6. `NAMING AND COUNTS`: naming-convention adherence and verified counts.
7. `SUMMARY AND PROVENANCE`: summary tables, manifest, packaging operator, time, and tool identity.
8. `TEMPORARY AND HIDDEN FILE CHECKS`: results of inspection for `.tmp`, `.part`, `.DS_Store`, dotfile directories, and similar.
9. `PRIVATE PATH AND METADATA CHECKS`: text-content inspection for absolute paths, host names, or environment-specific strings.
10. `README AND CLARITY`: presence, scope, audience, contents, and unresolved-item statements.
11. `PACKAGING COMMAND REPRODUCIBILITY`: command or script, inputs, regeneration evidence, and manual-step inventory.
12. `VALIDATION EVIDENCE`: commands actually run, exit statuses, concise results, checks not run, and impact.
13. `FINDINGS`: blocking findings and non-blocking warnings with location, evidence, consequence, and minimum resolution; write `none` for empty categories.
14. `NEXT STEP`: the minimum correction, regeneration, redaction, or authorization supported by the verdict.

## Allowed actions

- Read the source artifact set, the review-copy area, the packaging command or script, and the deliverable contents within scope.
- Run local read-only inspection commands (file listing, archive listing, name matching, count enumeration) when explicitly safe and authorized.
- Propose layout, naming, or summary corrections without applying them unless separately authorized.

## Forbidden actions

- Modify source artifacts during assembly or audit.
- Share, transmit, or transfer the deliverable without separate explicit authorization.
- Include `.DS_Store`, `Thumbs.db`, editor temporary files, dotfile directories, or credential-bearing files unless project policy explicitly requires them.
- Include absolute paths, host names, or environment-specific strings in text-visible deliverable content unless audit-material policy explicitly requires retention.
- Treat a one-off manual assembly as reproducible without an inspectable command.
- Expose private paths, restricted identifiers, or sensitive content in reports or commands.

## Verdict vocabulary

- `DELIVERABLE_READY`: Inventory, source separation, archive layout, naming, counts, summary, provenance, temporary/hidden-file checks, private-path checks, README clarity, and packaging reproducibility satisfy the selected mode with no remaining warnings.
- `DELIVERABLE_READY_WITH_WARNINGS`: No blocking finding remains, but substantiated non-blocking limitations or residual risks are explicitly reported.
- `NEEDS_REVISION`: One or more correctable blocking inventory, separation, layout, naming, or reproducibility findings prevent acceptance.
- `DELIVERABLE_UNSAFE`: Source modification, embedded private paths, unsafe archive content, or unverifiable regeneration make the deliverable unsafe to share.
- `BLOCKED`: Missing inputs, inaccessible evidence, unsafe inspection conditions, or unavailable essential validation prevent a reliable conclusion.
- `OUT_OF_SCOPE`: The requested packaging, audit, or deliverable component exceeds authorized scope.

When several verdicts apply, use `OUT_OF_SCOPE` for unauthorized scope, `BLOCKED` when the audit itself cannot proceed safely, `DELIVERABLE_UNSAFE` for demonstrated unsafe content, and `NEEDS_REVISION` for correctable blocking findings. Use a ready verdict only when the inspected evidence supports each packaging requirement.
